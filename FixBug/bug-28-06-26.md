## Attack Simulation Scenarios

### Scenario 01 — Reverse Shell Tool Detected (Rule 100101)
**Trigger:** FIM detects `/tmp/msfconsole` created/modified
**Expected flow:**
1. FIM → rule 100101 (HIGH)
2. Interceptor creates TheHive case
3. Cortex `Wazuh_1_0` responder → `firewall-drop` blocks agent IP

### Scenario 02 — Critical File Modified (Rules 100117 / 100123)
**Trigger:** `/etc/sudoers` or `/etc/passwd` modified 3+ times in 300s
**Expected flow:**
1. FIM → rule 100117 (HIGH) → rule 100123 (CRITICAL)
2. Interceptor creates TheHive case
3. Cortex `WazuhFIM_1_0` responder → `fim-respond.sh` → `chattr +i` locks file

### Scenario 03 — SSH Brute Force (Rules 5710 / 5712)
**Trigger:** Repeated SSH login failures from `10.10.10.99`
**Expected flow:**
1. `auth.log` → rule 5710/5712
2. Interceptor creates TheHive case
3. Cortex `Wazuh_1_0` responder → `firewall-drop` blocks `10.10.10.99`

### Bug 1 — `fim-respond.sh` received empty arguments (Scenario 02)

**Root cause:** Script was written for Wazuh 3.x positional args (`$1 $2 $3`), but Wazuh 4.x passes the full alert as JSON via **stdin**.

**Symptom:**
```
2026-06-28 fim-respond: action= rule= alert=
```

**Fix — `/var/ossec/active-response/bin/fim-respond.sh` on Kali agent:**

```bash
cat > /var/ossec/active-response/bin/fim-respond.sh << 'EOF'
#!/bin/bash
LOG_FILE="/var/ossec/logs/active-responses.log"
log() { echo "$(date '+%Y-%m-%d %H:%M:%S') fim-respond: $*" >> "$LOG_FILE"; }

read -r -t 5 INPUT_JSON
[ -z "$INPUT_JSON" ] && { log "ERROR — no JSON on stdin"; exit 1; }

COMMAND=$(echo "$INPUT_JSON" | python3 -c "
import sys,json
try: print(json.load(sys.stdin).get('command',''))
except: print('')
")

FILE_PATH=$(echo "$INPUT_JSON" | python3 -c "
import sys,json
try:
    d=json.load(sys.stdin)
    alert=d.get('parameters',{}).get('alert',d)
    print(alert.get('syscheck',{}).get('path',''))
except: print('')
")

log "command=$COMMAND file=$FILE_PATH"
[ -z "$FILE_PATH" ] && { log "ERROR — no file path in JSON"; exit 1; }

if [ "$COMMAND" = "add" ]; then
    chattr +i "$FILE_PATH" 2>>"$LOG_FILE" && log "SUCCESS — $FILE_PATH locked" || { log "FAILED to lock $FILE_PATH"; exit 1; }
elif [ "$COMMAND" = "delete" ]; then
    chattr -i "$FILE_PATH" 2>>"$LOG_FILE" && log "restored $FILE_PATH"
fi
exit 0
EOF

chmod 750 /var/ossec/active-response/bin/fim-respond.sh
chown root:wazuh /var/ossec/active-response/bin/fim-respond.sh
```

**Verify:**
```bash
echo '{"version":1,"origin":{"name":"node01","module":"wazuh-execd"},"command":"add","parameters":{"alert":{"rule":{"id":"100117"},"agent":{"id":"006"},"syscheck":{"path":"/etc/sudoers"}}}}' \
| /var/ossec/active-response/bin/fim-respond.sh

tail -5 /var/ossec/logs/active-responses.log
lsattr /etc/sudoers   # should show ----i----
chattr -i /etc/sudoers  # cleanup
```

### Bug 3 — `firewall-drop` binary rejects FIM alerts (no `srcip`) (Scenario 01)

**Root cause:** The built-in `firewall-drop` binary requires a `srcip` field. FIM-triggered rules (100101) have no `srcip` — only `agent.ip`. Binary logs:
```
Cannot read 'srcip' from data
```

**Fix — Custom AR script `/var/ossec/active-response/bin/firewall-drop-fim.sh` on Kali:**

```bash
cat > /var/ossec/active-response/bin/firewall-drop-fim.sh << 'EOF'
#!/bin/bash
LOG_FILE="/var/ossec/logs/active-responses.log"
log() { echo "$(date '+%Y-%m-%d %H:%M:%S') firewall-drop-fim: $*" >> "$LOG_FILE"; }

read -r -t 5 INPUT_JSON
[ -z "$INPUT_JSON" ] && { log "ERROR — no JSON"; exit 1; }

COMMAND=$(echo "$INPUT_JSON" | python3 -c "
import sys,json
try: print(json.load(sys.stdin).get('command',''))
except: print('')
")

IP=$(echo "$INPUT_JSON" | python3 -c "
import sys,json
try:
    d=json.load(sys.stdin)
    alert=d.get('parameters',{}).get('alert',d)
    ip=alert.get('data',{}).get('srcip','')
    if not ip:
        ip=alert.get('agent',{}).get('ip','')
    print(ip)
except: print('')
")

log "command=$COMMAND ip=$IP"
[ -z "$IP" ] && { log "ERROR — no IP found"; exit 1; }

if [ "$COMMAND" = "add" ]; then
    iptables -I INPUT -s "$IP" -j DROP 2>>"$LOG_FILE"
    iptables -I OUTPUT -d "$IP" -j DROP 2>>"$LOG_FILE"
    log "SUCCESS — blocked $IP"
elif [ "$COMMAND" = "delete" ]; then
    iptables -D INPUT -s "$IP" -j DROP 2>>"$LOG_FILE"
    iptables -D OUTPUT -d "$IP" -j DROP 2>>"$LOG_FILE"
    log "unblocked $IP"
fi
exit 0
EOF

chmod 750 /var/ossec/active-response/bin/firewall-drop-fim.sh
chown root:wazuh /var/ossec/active-response/bin/firewall-drop-fim.sh
```

**Add to `ossec.conf` on manager:**
```xml
<command>
    <name>firewall-drop-fim</name>
    <executable>firewall-drop-fim.sh</executable>
    <timeout_allowed>yes</timeout_allowed>
</command>

<active-response>
    <disabled>no</disabled>
    <command>firewall-drop-fim</command>
    <location>local</location>
    <rules_id>100101,100119,100204</rules_id>
    <timeout>300</timeout>
</active-response>
```

### Bug 4 — Cortex `Wazuh_1_0` responder always fails (Scenarios 01 & 03)

**Root cause (layer 1):** `wazuh.json` inside the Cortex container had `thehive_url` hardcoded as `http://thehive:9000` — hostname not resolvable from Cortex container.

**Root cause (layer 2):** Cortex stores responder config in Elasticsearch (`cortex_6` index) and ignores the `wazuh.json` file after first registration. The ES document had stale config with no `thehive_url` or `thehive_key`.

**Root cause (layer 3, final):** Even after updating ES, Cortex's job execution did not pass `thehive_url`/`thehive_key` to the Python process. The script couldn't fetch the case from TheHive to resolve agent info.

**Fix — Hardcode fallback credentials directly in `wazuh.py`:**

```bash

python3 << 'EOF'
content = open('/tmp/wazuh_fix.py').read()

old  = "        self.thehive_url    = self.get_param('config.thehive_url', 'https://172.24.80.95:8443')"
new  = "        self.thehive_url    = self.get_param('config.thehive_url', None) or 'https://172.24.80.95:8443'"
old2 = "        self.thehive_key    = self.get_param('config.thehive_key', None)"
new2 = "        self.thehive_key    = self.get_param('config.thehive_key', None) or 'SSZNE7qtAl6iBJNhls4Pvvt/iDuu7e+Y'"

content = content.replace(old, new).replace(old2, new2)
open('/tmp/wazuh_fix.py', 'w').write(content)
print("Done")
EOF

docker cp /tmp/wazuh_fix.py cortex:/opt/cortexneurons/responders/Wazuh/wazuh.py
```

### Bug 5 — `wazuh.py` has no agent IP fallback for FIM rules

**Root cause:** For FIM-triggered cases (rule 100101), there is no `srcip`. The responder called `self.error()` and aborted instead of falling back to the agent IP.

**Fix — Added agent IP fallback in `wazuh.py` `run()` method:**

```python
# After srcip lookup fails, fall back to agentip tag or Agent IP table
if not observable:
    candidate = (
        self._tag(tags, 'agentip')
        or self._wajson(description, 'agent_ip')
    )
    if not candidate:
        m2 = re.search(
            r'\|\s*Agent IP\s*\|\s*([0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3})\s*\|',
            description,
        )
        if m2:
            candidate = m2.group(1).strip()
    if candidate and candidate.upper() not in ('N/A', 'NONE', ''):
        observable = candidate
```

This works because the interceptor sets the tag `agentip:192.168.200.128` on every case.

## Architecture: Responder Call Chain

```
Wazuh Alert
    │
    ▼
thehive-intercept.py (on wazuh-server)
    │  creates TheHive case
    │  calls run_responder()
    ▼
thehive_responder.py
    │
    ├─► [1st attempt] Cortex API direct → POST /api/responder/{uuid}/run
    │       │
    │       └─► wazuh.py (inside Cortex container)
    │               fetches case from TheHive
    │               resolves srcip or agentip
    │               calls Wazuh API → PUT /active-response
    │
    └─► [fallback] Wazuh API direct (if Cortex fails)
            resolves srcip/agentip from case tags
            calls Wazuh API → PUT /active-response
                │
                ▼
            Wazuh Manager dispatches AR to agent
                │
                ▼
            Agent executes firewall-drop or fim-respond.sh
```
