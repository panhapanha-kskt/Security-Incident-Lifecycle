# Fixing Wazuh Agent Startup Timeout

## Overview

This document explains the troubleshooting process for resolving a Wazuh Agent startup timeout on a Kali Linux machine. Although the Wazuh service initially reported a startup failure, the agent processes were actually running in the background. The issue was caused by a temporary startup timeout rather than a network or configuration problem.

---

# Environment

| Component | Version |
|-----------|----------|
| Operating System | Kali Linux |
| Wazuh Agent | 4.14.5 |
| Wazuh Manager | 4.14.5 |
| Manager IP | 192.168.200.129 |

---

# Problem

Running the following command:

```bash
systemctl status wazuh-agent
```

returned:

```text
Active: inactive (dead)

wazuh-agent.service: start operation timed out.
Failed with result 'timeout'.
```

However, executing:

```bash
/var/ossec/bin/wazuh-control status
```

showed:

```text
wazuh-modulesd is running...
wazuh-logcollector is running...
wazuh-syscheckd is running...
wazuh-agentd is running...
wazuh-execd is running...
```

This indicated that the Wazuh daemons had started successfully even though the systemd service reported a timeout.

---

# Symptoms

- `systemctl` reported a timeout.
- Wazuh processes continued running.
- Agent successfully connected to the manager.
- Manager recognized the agent as Active.
- Startup script temporarily failed to exit before systemd timeout.

---

# Troubleshooting Steps

## 1. Verify Wazuh Processes

```bash
ps -ef | grep wazuh
```

Expected output:

```text
/var/ossec/bin/wazuh-agentd
/var/ossec/bin/wazuh-syscheckd
/var/ossec/bin/wazuh-logcollector
/var/ossec/bin/wazuh-modulesd
/var/ossec/bin/wazuh-execd
```

---

## 2. Check Wazuh Service Status

```bash
systemctl status wazuh-agent
```

Initial result:

```text
Active: inactive (dead)
start operation timed out
```

---

## 3. Verify Wazuh Daemons

```bash
/var/ossec/bin/wazuh-control status
```

Output:

```text
wazuh-modulesd is running...
wazuh-logcollector is running...
wazuh-syscheckd is running...
wazuh-agentd is running...
wazuh-execd is running...
```

---

## 4. Check Agent Connectivity

```bash
grep --text "Connected to the server" /var/ossec/logs/ossec.log | tail
```

Output:

```text
Connected to the server ([192.168.200.129]:1514/tcp)
```

This confirms successful communication with the Wazuh Manager.

---

## 5. Verify Agent Status from Manager

On the Wazuh Manager:

```bash
/var/ossec/bin/agent_control -l
```

Output:

```text
ID: 006
Name: Agent-Kali
Status: Active
```

This confirms that the manager successfully receives heartbeats from the Kali agent.

---

# Resolution

Restart the Wazuh service:

```bash
systemctl restart wazuh-agent.service
```

Verify the service:

```bash
systemctl status wazuh-agent.service
```

Expected output:

```text
Active: active (exited)

Completed.
```

This status is normal because the systemd service launches the Wazuh daemons and then exits while the daemons continue running in the background.

---

# Verification

Check daemon status:

```bash
/var/ossec/bin/wazuh-control status
```

Expected:

# Warnings Observed

The following warnings were present but did not prevent the agent from operating:

## Missing audit socket

```text
Cannot connect to socket 'queue/sockets/audit'
```

Reason:

- auditd service is not enabled.

Impact:

- Wazuh automatically switches to realtime monitoring.

---

## Missing syslog

```text
Could not open file '/var/log/syslog'
```

Reason:

- Kali Linux primarily uses systemd-journald instead of `/var/log/syslog`.

Impact:

- Safe to ignore.

---

## Missing Suricata log

```text
Could not open file '/var/log/suricata/eve.json'
```

Reason:

- Suricata is not installed or not running.

Impact:

- No effect unless Suricata integration is required.

---

## Queue Full Warning

```text
Target 'agent' message queue is full (1024)
```

Reason:

- Temporary backlog during startup.

Impact:

- Usually harmless if it only occurs once during initialization.

---

# Root Cause Analysis

The issue was **not caused by:**

- Internet connectivity
- Firewall configuration
- Manager availability
- Agent registration
- Agent authentication

The evidence indicates a temporary startup timeout where:

1. The Wazuh daemons started successfully.
2. The startup script (`wazuh-control`) temporarily hung.
3. systemd reached its timeout limit.
4. Restarting the service allowed the startup script to complete normally.

# Conclusion

The startup timeout was a transient issue rather than a configuration or networking problem. After restarting the Wazuh Agent service, all Wazuh daemons started correctly, the service completed successfully, and the Wazuh Manager recognized the Kali agent as **Active**. The monitoring environment is now functioning as expected.
